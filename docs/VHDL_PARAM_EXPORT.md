# VHDL Parametric Export — Design Notes

> **Status:** Design decided, implementation pending.
> Planned as a separate utility project (`spinalhdl-vhdl-export`) forked from the template.
> This document captures the decisions made so implementation can start cleanly.

---

## Problem Statement

SpinalHDL elaborates all parameters at JVM time before generating any HDL. By the time
`generateVhdl` runs, every signal has a concrete bit-width and every conditional has been
evaluated. The output is always a fully-concrete VHDL entity — no generics, no `generate`
statements, no parameterisation.

This creates a friction point when delivering IP to a VHDL team whose existing designs use
top-level constants or generics that propagate down through the hierarchy. A SpinalHDL-generated
entity becomes a dead end in that chain — they cannot tweak `G_DATA_WIDTH` at their top level
and have the change flow into the IP.

---

## Why the Limitation Exists (and Why It Is Acceptable)

The same elaboration model that prevents generic emission is what enables:

- Compile-time width inference and type safety
- `require()` guards that catch bad parameters before any RTL is written
- Fiber dependency resolution across plugins
- The entire `Params` / `FiberPlugin` / service trait reconfigurability pattern

This is a deliberate SpinalHDL design choice, not an oversight. The trade-off is accepted.
The goal of this utility is to **mitigate the integration friction** for VHDL consumers, not
to circumvent the elaboration model.

---

## Solution Overview

Three-step generation pipeline, all in Scala, all triggered by one `sbt runMain` command:

```
Step 1: SpinalHDL generates MyTop.vhd          (concrete entity, unchanged)
Step 2: GenVhdl writes MyTop_pkg.vhd           (VHDL package of named constants)
Step 3: VhdlConstantInjector rewrites MyTop.vhd (internal signals → constant references)
```

VHDL consumers receive two files:
- `MyTop.vhd` — entity with concrete port widths, architecture using named constants internally
- `MyTop_pkg.vhd` — package of all Params values as named VHDL constants

Entity port widths remain concrete literals (unavoidable). All **internal** signal widths
reference the package constants. One documented manual step promotes any constant to a generic
if the consumer needs live parameter propagation from their hierarchy.

## Architecture

### `VhdlExportable` Trait

A lightweight trait mixed into `Params`. Defines the mapping from Scala fields to VHDL
constant names. This is the single source of truth — if a field is renamed in Params,
the compiler flags the mismatch here before any bad output is generated.

```scala
trait VhdlExportable {
  def vhdlConstants: Seq[(String, String, Int)]
  // (VHDL constant name, description, value)
  // e.g. ("C_TIMER_WIDTH", "Timer counter width in bits", 8)

  def toVhdlPackage(entityName: String): String = {
    val body = vhdlConstants.map { case (name, desc, value) =>
      f"  constant $name%-24s : integer := $value%-6d;  -- $desc"
    }.mkString("\n")
    s"""|library ieee;
        |use ieee.std_logic_1164.all;
        |use ieee.numeric_std.all;
        |
        |-- AUTO-GENERATED by SpinalHDL GenVhdl — do not edit by hand.
        |-- To connect to a top-level generic, see the "Generics Upgrade Path"
        |-- section in VHDL_PARAM_EXPORT.md.
        |package ${entityName}_pkg is
        |$body
        |end package;""".stripMargin
  }
}
```

`Params` defines the mapping inline:

```scala
case class Params(
    timerWidth: Int = 8,
    threshold:  Int = 128
) extends VhdlExportable {

  def vhdlConstants = Seq(
    ("C_TIMER_WIDTH", "Timer counter width in bits",  timerWidth),
    ("C_THRESHOLD",   "Comparator threshold value",   threshold),
    ("C_APB_ADDR_W",  "APB3 address bus width",       8),
    ("C_APB_DATA_W",  "APB3 data bus width",          32)
  )
}
```

### `VhdlConstantInjector`

Post-processes the generated `.vhd` file in Scala. Replaces literal widths on internal
signals with references to the package constants, scoped by `periphName` prefix so
coincidental same-width signals in unrelated logic are not affected.

**Why `periphName` scoping is safe:**
The `periphName` convention (e.g., `timer_countReg`, `comparator_aboveReg`) means every
signal that belongs to a given parameter domain has a predictable prefix. A substitution
targeting `timer_\w+` will never incorrectly modify an APB address register.

```scala
object VhdlConstantInjector {

  def inject(vhdlPath: String, entityName: String, params: VhdlExportable): Unit = {
    val source  = scala.io.Source.fromFile(vhdlPath)
    var content = source.mkString
    source.close()

    // Inject use clause before the architecture
    content = content.replace(
      s"architecture rtl of $entityName",
      s"use work.${entityName}_pkg.all;\n\narchitecture rtl of $entityName"
    )

    // Replace literal widths on periphName-prefixed signals
    for ((constName, _, value) <- params.vhdlConstants if value > 1) {
      val periphPrefix = constName.stripPrefix("C_").takeWhile(_ != '_').toLowerCase
      val literalWidth = value - 1

      val pattern = (
        s"(signal\\s+${periphPrefix}_\\w+\\s*:\\s*(?:unsigned|signed)\\()" +
        s"$literalWidth(\\s+downto\\s+0\\))"
      ).r

      content = pattern.replaceAllIn(content,
        m => s"${m.group(1)}$constName-1${m.group(2)}  -- @PARAM $constName"
      )
    }

    val writer = new java.io.PrintWriter(vhdlPath)
    writer.write(content)
    writer.close()
  }
}
```

**`-- @PARAM C_TIMER_WIDTH` annotations** serve as machine-readable markers.
A future script (or a VHDL engineer doing a manual upgrade) can grep for `@PARAM` to find
every site that is parameterised and what constant it tracks.

### `GenVhdl.scala` Integration

```scala
object GenVhdl extends App {
  val params    = Params(timerWidth = 8, threshold = 128)
  val targetDir = "rtl"
  val topName   = "MyTop"

  // Step 1 — generate entity
  SpinalConfig(targetDirectory = targetDir)
    .generateVhdl(new MyTop(params))
    .printPruned()

  // Step 2 — write companion package
  val pkgPath = s"$targetDir/${topName}_pkg.vhd"
  new java.io.PrintWriter(pkgPath) { write(params.toVhdlPackage(topName)); close() }
  println(s"[GenVhdl] wrote $pkgPath")

  // Step 3 — inject constant references
  VhdlConstantInjector.inject(s"$targetDir/$topName.vhd", topName, params)
  println(s"[GenVhdl] constants injected into $targetDir/$topName.vhd")
}
```

## The Generics Upgrade Path (for VHDL consumers)

When a consumer needs to break the constant chain and connect a parameter to their own
top-level generic, the procedure is a one-time, three-site manual edit. Document this
alongside every IP release.

**Example: promoting `C_TIMER_WIDTH` to a generic**

**Site 1 — Add generic to entity port declaration:**
```vhdl
-- Before (generated):
entity MyTop is
  port( count : out unsigned(7 downto 0); ... );
end entity;

-- After (manual edit):
entity MyTop is
  generic( G_TIMER_WIDTH : integer := 8 );
  port( count : out unsigned(G_TIMER_WIDTH-1 downto 0); ... );
end entity;
```

**Site 2 — Replace constant with generic in architecture:**
```vhdl
-- Remove the use clause and constant reference:
-- use work.MyTop_pkg.all;
-- The constant C_TIMER_WIDTH is now a generic — all @PARAM references resolve to it.

-- Internal signals already use C_TIMER_WIDTH-1 from the injector step.
-- Rename C_TIMER_WIDTH → G_TIMER_WIDTH throughout, or declare an alias:
constant C_TIMER_WIDTH : integer := G_TIMER_WIDTH;
```

**Site 3 — Pass generic from their top level:**
```vhdl
inst_my_top : entity work.MyTop
  generic map( G_TIMER_WIDTH => SYSTEM_DATA_WIDTH )
  port map( ... );
```

Once done, their top-level constant propagates through the IP with no further SpinalHDL
involvement. The `-- @PARAM` annotations remain in place as documentation.

## Decision Log

### Decision: Scala, not Python

**Considered:** External Python script for post-processing generated VHDL.

**Rejected because:**
- Python is disconnected from the `Params` object — parameter values must be passed via CLI
  arguments or a separate config file, creating a second source of truth.
- If a Params field is renamed, the Python script silently generates wrong output. The Scala
  version fails to compile.
- Adds a second toolchain dependency (`python3`) alongside the existing `sbt` / `java`.
- The constant-name-to-field mapping is logic that belongs with the domain model (`Params`),
  not in an external script.

**Chosen:** All post-processing in Scala. `VhdlExportable` on `Params` is the single source
of truth. Running `sbt "runMain mydesign.GenVhdl"` produces all artifacts in one step.

Python remains acceptable for one-off transformations on third-party or legacy VHDL files
that are not generated by this project.

### Decision: Constant injection scoped by `periphName` prefix

**Rationale:** Naive width substitution (replace all `unsigned(7 downto 0)`) incorrectly
modifies unrelated signals that happen to share the same bit-width. The `periphName` naming
convention (already required by the SpinalHDL template) makes every signal's parameter domain
explicit in its name, enabling safe, targeted substitution.

**Consequence:** The `periphName` convention is now load-bearing for the export utility.
Any Core that does not follow the `setName(s"${periphName}_xxxReg")` convention will have
its internal signals left as literals after injection.

### Decision: Entity port widths remain concrete

**Rationale:** SpinalHDL cannot emit generic declarations on entity ports. Post-processing
the port declaration to inject a generic reference would produce a VHDL entity that does not
match the elaborated netlist — a synthesis mismatch.

Port widths are left as generated literals. The companion package documents what those widths
correspond to. The documented upgrade path is the mechanism for consumers who need live
port-width parameterisation.

## Hard Limits — What This Does Not Solve

| Scenario | Status |
|---|---|
| Consumer changes `G_DATA_WIDTH` at instantiation with no edits to generated files | **Not achievable** — requires the one-time manual generic promotion |
| Embedding SpinalHDL IP inside a fully parameterised VHDL hierarchy with no manual step | **Not achievable** — fundamental elaboration model constraint |
| Generating VHDL `generate` blocks or conditional architecture sections | **Not achievable** from SpinalHDL output |
| Different internal logic topology for different widths from one VHDL file | **Not achievable** — generate multiple named variants instead |

## Planned Project Structure

To be implemented as `spinalhdl-vhdl-export` — a small library extracted from the template
so it can be shared across projects.

```
spinalhdl-vhdl-export/
├── build.sbt                          ← minimal deps: Scala std lib only, no SpinalHDL
├── src/main/scala/vhdlexport/
│   ├── VhdlExportable.scala           ← trait + toVhdlPackage()
│   └── VhdlConstantInjector.scala     ← post-processor
└── src/test/scala/vhdlexport/
    └── VhdlConstantInjectorTest.scala ← unit tests against fixture VHDL strings
```

Any SpinalHDL project adds one line to `build.sbt`:
```scala
"com.yourcompany" %% "spinalhdl-vhdl-export" % "1.0.0"
```

And `Params` gains `extends VhdlExportable` with a `vhdlConstants` definition.
The `VhdlConstantInjector` has no SpinalHDL dependency — it operates on strings — so the
library is usable from any Scala project, not just SpinalHDL ones.
